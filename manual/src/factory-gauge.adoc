= Factory Gauge
:xrefstyle: short

Factory gauges are used for automating logistic requests and production, for their conventional usage and foundational concepts of higher logistics, see the in-game ponder. When placed in a factory controller, they operate in recipe mode.

.A new factory gauge placed in a factory controller
image::gauge-empty.png[,70,67]

[#place-set-item]
== Place and Monitor Item

Factory gauges need to be tuned to a logistics network before being placed in a factory controller. To do so, either interact with a stock link in the game world with the gauge in hand, or use the xref:dashboard.adoc#network-selector[network selector] in dashboard UI to tune to a network that is already tuned to by another gauge in the factory controller.

Once placed, the new factory gauge needs to be configured with an item to monitor. You may do so by clicking on the unconfigured gauge with the item on the cursor, or by using the "set item" screen. Either way, you may also drag a ghost item from JEI or EMI to the filter slot.

[#figure-set-item]
."Set item" screen
image::gauge-set-item-screen.png[,344,174]

If you use the "set item" screen, you can choose whether to respect image:btn-respect-nbt.png[,18,18] or ignore image:btn-ignore-nbt.png[,18,18] the item's NBT data. NBT is respected by default.

Once an item is set, the gauge starts monitoring the number of these items in the tuned logistic network and displays the count on its label.

.A factory gauge monitoring iron ingots, for which there are 6
image::gauge-monitoring-item.png[,68,68]

[#basic-production]
== Basic Production

Beyond monitoring storage, the most basic usage of a factory gauge is to represent a recipe to produce an item, for example, smelting 1 raw iron into 1 iron ingot. The type of the output is simply the monitored item. To define the ingredient(s), connect this gauge with other factory gauges set to monitor those ingredients.

This connection can be made by clicking "add connection" (<<figure-iron-smelting-recipe>> ❹), and then the gauge responsible for the ingredient. Such connections are shown as item icons in the ingredient list (❷). They can be removed by holding shift and clicking on them.

.A gauge monitoring raw iron is connected to a gauge set to produce iron ingots
image::gauge-ingredient-connection.png[,200,68]

[#figure-iron-smelting-recipe]
.Recipe settings of a gauge smelting 1 raw iron to 1 iron ingot
image::gauge-iron-smelting-recipe.svg[,384,368]

In its default mode (⓬), a factory gauge monitors the amount of its item (❾) in the logistic network storage it is tuned to. When this amount is below the set target (❿⓫), it requests a set of ingredients (❷) to be sent to the destination address (❽), expecting the output (❶) to later arrive in storage (❾).

Both the number of each ingredient and output can be adjusted by scrolling on them, allowing a configurable number of output items to be produced from configurable numbers of ingredients. Scrolling on an ingredient allows you to configure more than 1 stack of the same item. However, there cannot be more than 9 stacks of ingredients in total, the output is also limited to 9 stacks.

.Examples
[%autowidth,cols="2*"]
|===
| Smelting raw iron to ingots in 32-item batches
| image:gauge-batch-smelting-example.png[,251,154]

| Mixing concrete powder in 256-item batches
| image:gauge-batch-mixing-example.png[,251,154]

When a factory gauge A is used as an ingredient for another gauge B, most of A's configurations are irrelevant to B, except for the item A is set to monitor and the network A is tuned to. This means A can produce its item by any means or merely monitor it, B treats both cases identically.

[#crafting]
== Mechanical Crafting

When the ingredients and output match a mechanical crafting recipe, a "use mechanical crafting" button (<<figure-crafting-1>> ❶) is revealed. With this option toggled on, when packages created by this gauge are unpacked onto mechanical crafters, the ingredients are automatically arranged into the crafting pattern (❷).

[#figure-crafting-1]
.Crafting 1 iron sword
image::gauge-crafting-iron-sword.svg[,340,154]

The crafting recipe may be at most 10×10 in size. However, when it is larger than 3×3, ❷ no longer previews the pattern, instead hold ctrl while hovering to reveal a preview tooltip. Factory gauges that ignore NBT data may not be used as ingredients for crafting recipes larger than 3×3. If the ingredients of a large crafting recipe cannot be packed within 9 stacks, ❶ is not shown.

.Crafting crushing wheels
image::gauge-crafting-crushing-wheels.png[,340,154]

The mechanical crafters must be connected using a wrench and arranged in a square grid between 3×3 and 10×10 in size. Smaller recipes may be sent to larger mechanical crafter arrangements. Hold ctrl and scroll on the ingredients to adjust the crafting grid size to match the crafter arrangement they will be sent to.

It is recommended to use a repackager before unpacking onto mechanical crafters, see the in-game ponder on repackagers.

[#batch-crafting]
=== Batch Crafting

A factory gauge may be configured to send ingredients enough to craft multiple output items at once. When mechanical crafting is turned on, scroll on the output slot (<<figure-crafting-batch>> ❶) to adjust how many items to craft each batch. The number of ingredients (❷) are matched automatically and cannot be adjusted individually.

[#figure-crafting-batch]
.Crafting iron swords in 64-item batches
image::gauge-crafting-batch.svg[,340,154]

Using a repackager is mandatory when batch crafting is used. When a repackager receives a package containing ingredients for multiple crafts, it is repackaged into multiple smaller packages containing ingredients for 1 craft each, which may then be unpacked onto mechanical crafters.

The maximum batch size is 64, regardless of the stack size of the output item. This means for a crafting recipe that produces N output items per craft, the maximum number of output items is 64 × N.

.Crafting crushing wheels in 128-item batches
image::gauge-crafting-batch-crushing-wheels.png[,340,154]

When using batch crafting, the total number of ingredients in a request may exceed 9 stacks, in which case the ingredients are split into multiple packages. A repackager awaits all of them, and repackages them into smaller packages.

If an ingredient of a batch crafting recipe comes from a factory gauge that ignores item NBT data, and the recipe requires X amount of this ingredient, the ingredient will only be sent when there has accumulated X amount of items with the same NBT data.

[#ingredient-arrange]
== Specify Ingredient Arrangement

By default, when packages created by factory gauges are unpacked, the contents are not arranged in any specific order. Clicking <<figure-arrange-deployer>> ❶ enables custom arrangement, allowing you to assign each ingredient to any of the 9 slots. This is similar to <<crafting,mechanical crafting>>, but the arrangements may be arbitrary, and not limited crafting recipes.

[#figure-arrange-deployer]
.Using a custom arrangement to represent a deploying recipe
image::gauge-arrange-deployer.svg[,340,154]

When custom arrangement is enabled, each ingredient in ❷ may be dragged with the left mouse button to be moved to another slot. Drag with the right mouse button to duplicate the ingredient. Neither

Using a repackager is required for the custom arrangement to take effect, even if one package already contains a complete set of ingredients. After repackaging, the repackaged package can then be unpacked into any container block. Ingredients are unpacked into their assigned slot(s). In the example given by <<figure-arrange-deployer>>, the sandpapers are unpacked into the 1st-3rd slots of the container, and the rose quartz are unpack into the 9th slot. The receiving end may then be set up to transfer the items in the 1st-3rd slots onto a deployer, and the items in the 9th slot under the deployer.

.Contents of a package created by <<figure-arrange-deployer>> unpacked into an empty chest
image::gauge-arrange-deployer-unpacked.png[,352,76]

For unpacking to be possible, each item stack specified in ❷ must be unpacked into each of the first 9 slots of the target container. It may stack with existing items in the slot, if they are stackable and do not exceed the stack size. Item stacks do not overflow to another slot. The target container may have less than 9 slots, if the custom arrangement do not make use of the slots beyond what is available in the container.

A package with custom arrangement may be unpacked into any block supported by packagers. The contents are unpacked according to the number and order of slots made available by the target block. For example, such packages may be unpacked directly onto a vanilla crafter to perform batch crafting; but a package containing both a smeltable item and a fuel may not be unpacked onto a furnace, because a packager may only access one of the slots in the furnace depending on the side it is attached to.

[#on-demand]
== On-Demand Production

By default, a factory gauge operates in image:btn-maintain-stock.png[18,18] "maintain stock" mode, which activates (requests ingredients to be sent to the destination) when the monitored item in storage is below the configured target amount. By clicking <<figure-follow-demand>> ❷, you may switch it to image:btn-follow-demand.png[,18,18] "follow demand" mode, which instead activates the gauge when there is downstream demand, that is, when another gauge uses this gauge as an ingredient, and that gauge would like to request this ingredient.

[#figure-follow-demand]
.A gauge producing oak planks in "follow demand" mode
image::gauge-follow-demand-mode.svg[,296,48]

In this mode, the target amount (❶) cannot be modified manually. It is automatically set according to downstream demand. Specifically, it is the total quantity requested by all currently active factory gauges that use this gauge as an ingredient.

.An example setup to produce barrels from oak logs
image::gauge-follow-demand-barrel-example.png[,329,198]

For example, to produce barrels, one may use 6 oak planks and 2 oak slabs. 6 slabs may be crafted from 3 planks, and 4 planks may be crafted from 1 log. The planks and slabs can be produced with gauges in "follow demand" mode:

* The gauge for slabs receives a demand of 2.
* The gauge for planks receives a demand of 9: 6 from barrels, 3 from slabs.
* First, 3 logs are crafted into 12 planks, then 3 of which are crafted into 6 slabs.
* After crafting 1 barrel, 3 planks and 4 slabs are left over.

[#production-orders]
=== Ordering from Stock Keeper

Another mode of the factory gauge, image:btn-follow-demand-and-order.png[,18,18] "follow demand and order", additionally allows the item produceable by a gauge to be manually ordered from a stock keeper tuned to the same network. In stock keeper UI, each gauge in this mode appears as a production task, signified by a blueprint background.

.A production task of oak planks
image::gauge-production-task.png[,308,111]

See xref:production-orders.adoc[Production Orders] for details.

[#fluid]
== Fluid Transport

When you have a compatible fluid transportation mod installed, factory gauges placed in factory controllers may utilize that mod's ability to transport fluids, and use fluids as ingredients or outputs.

.Compatibility of fluid transportation mods
[%header%autowidth,cols="2*"]
|===
| Mod | Compatibility

| Create: FluidLogistic
| Yes

| Create: Fluid
| Yes, but may drop support in the future

| Create: Repackaged
| Yes

| Create Factory Logistics
| No, will crash the game
|===

When using Create: Repackaged, place fluid gauges provided by this mod to monitor and produce fluids. Fluids may be set in the <<figure-set-item,"set item" screen>> by clicking the filter slot while holding a bucket of fluid or by dragging a fluid entry from JEI.

When using the other mods, regular factory gauges gain the ability to monitor and produce fluids. Fluids may be set in the <<figure-set-item,"set item" screen>> by right-clicking the filter slot while holding a bucket of fluid or by dragging a fluid entry from JEI.

Fluids do not contain NBT data, therefore when monitoring a fluid, the option to ignore NBT data disappears.

When using a fluid as an ingredient or output, scrolling on the ingredient or output UI adjusts the amount by 1 B. Hold shift while scrolling to adjust by 10 mB, or hold ctrl while scrolling to adjust by 1 mB. The output is limited to 10 B per batch.

[#promise]
== Promise Limits

Each time a factory gauge requests a set of ingredients to be sent to its destination address, a promise is created (also referred to as a pending request). Each promise records the number of expected to be produced, but have yet to arrive in storage. When accounting for the number of monitored items in storage, the number of items recorded in promises is added to the actual storage amount before being compared to the target amount. Doing so stops the factory gauge from excessively sending ingredients before the outputs of previous requests arrive.

.UI elements related to promises
image::gauge-promise-ui.svg[,266,52]

❶ Promised items::
Displays the total number of items recorded in promises created by all gauges in the same network producing the same item. Click to void all promises of the same item type.

❷ Expire time::
Scroll to set the time interval for a promise to expire after its creation. Can be 1 to 30 minutes. Promises never expire by default. +
  This option is useful when the ingredients do not reliably produce the expected output, such as with recipes with probabilities.

❸ Pending requests::
Displays the number of pending requests created by this gauge image:symbol-gauge.png[,18,16]. Click to switch to displaying pending requests sent to the same address image:symbol-address.png[,18,16]. In either state, scroll to set a limit on the displayed number. When the limit is reached, the gauge pauses sending new requests. +
  Limiting pending requests is useful for preventing excessive packages from overloading transportation lines.

[#redstone]
== Redstone

A factory gauge can be connected to some other components, such as xref:electron-tube.adoc[logical tubes] and xref:redstone-link.adoc[redstone links], via redstone connections. When connected as the source, it outputs a redstone signal when the amount of the monitored item in storage is equal to or greater than the target amount. When connected as the sink, the factory gauge is disabled when it is powered by redstone.

.A factory gauge disabled by an incoming redstone signal, and outputting a redstone signal
image::gauge-redstone-io.png[,167,203]

Additionally, when a factory controller block is redstone-powered, all factory gauges inside are disabled.
